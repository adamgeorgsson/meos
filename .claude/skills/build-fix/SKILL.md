# Build Fix Skill

Fix build errors and trace them back to the skill or PRD that generated the broken code, updating it so the error won't recur in future runs.

## Step 1: Get the error log

If the user did not provide the build error output, ask them:

> Please paste the build error log (compiler/linker output).

## Step 2: Analyze errors

Parse each error from the log. Extract:

- **File path** and **line number**
- **Error type** (compiler error, linker error, warning-as-error)
- **Symbol** (undefined reference, undeclared identifier, type mismatch, etc.)
- **Root cause** (missing include, wrong signature, missing definition, type error, etc.)

Group related errors — a single root cause often produces multiple cascading errors. Fix root causes first.

## Step 3: Fix the code

For each root cause:

1. Read the affected file(s) around the error location.
2. Apply the **minimal fix** — don't refactor, don't clean up surrounding code.
3. Verify the fix doesn't introduce new issues (check types, includes, forward declarations).

## Step 4: Trace back to the originating skill or PRD

For each fix, determine **what generated the broken code**:

1. **Check git blame** on the broken line — find the commit that introduced it.
2. **Match commit message to skill/PRD** — commit messages reference user story IDs (e.g., `US-014`, `US-P0f2`) and skill names.
3. **Search skills** — grep `.claude/skills/*/SKILL.md` for mentions of the affected file, function, or pattern.
4. **Search PRDs** — grep `plan/prd-*.md` for the user story ID from the commit message.

If the error is in code that was NOT generated by a skill/PRD (i.e., it's original legacy code), skip this step for that error.

## Step 5: Update the skill or PRD

Add the learning to prevent the error from recurring if the task is re-executed from scratch.

### For skills (`.claude/skills/*/SKILL.md`):

Add the learning to a `## Known pitfalls` section. Create the section if it doesn't exist. Place it at the end of the skill, before any closing `---`.

Format:
```markdown
## Known pitfalls

- **<short description>**: <what went wrong and how to avoid it>
```

If a `## Known pitfalls` section already exists, append the new bullet point. Deduplicate — don't add if an equivalent learning already exists.

### For PRDs (`plan/prd-*.md`):

Add the learning under the relevant user story's `**Learnings from Previous Runs:**` section. Create the section if it doesn't exist (place after `**Implementation Notes:**` or `**Acceptance Criteria:**`).

Format:
```markdown
**Learnings from Previous Runs:**
- <what went wrong and how to avoid it>
```

Deduplicate against existing learnings and implementation notes.

## Step 6: Summary

After all fixes are applied and skills/PRDs are updated, provide a summary:

| Error | File | Fix | Updated skill/PRD |
|-------|------|-----|--------------------|
| ... | ... | ... | ... |

## Checklist

Before finishing:

- [ ] All build errors from the log are fixed
- [ ] Each fix is minimal — no unnecessary refactoring
- [ ] For each fix caused by skill/PRD-generated code, the source skill or PRD has been updated
- [ ] No duplicate learnings were added
- [ ] Summary table is provided to the user
- [ ] All changes are committed with a descriptive commit message referencing the fixed errors
